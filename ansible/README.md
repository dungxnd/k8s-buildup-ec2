# ansible/ — Configure the OS

Ansible takes bare Ubuntu servers and makes them Kubernetes-ready. It's agentless: connects via SSH to every machine in `inventory.ini` and runs tasks. It's idempotent: run it once or ten times — the result is the same.

Ansible reads `inventory.ini` to find its targets. That file is auto-generated by Terraform during `terraform apply`.

---

## File-by-file deep dive

### `setup.yml`

One playbook, one play, targeting every host. Let's go task by task.

**Play header:**

```yaml
---
- name: Prepare Ubuntu nodes for Kubernetes
  hosts: all
  become: true
```

- `---` marks the start of YAML.
- `hosts: all` targets every host in the inventory — master and workers get identical setup.
- `become: true` runs all tasks as root (via sudo). Installing packages, loading kernel modules, and writing config files under `/etc/` all require root.

**Play-level variables:**

```yaml
  vars:
    kubernetes_version: "1.36"
    kubernetes_apt_version: "v{{ kubernetes_version }}"
```

`kubernetes_apt_version` evaluates to `v1.36`. The Kubernetes APT repository uses a `v` prefix in its URL structure, so this avoids repeating the version string.

#### Task 1: Disable swap

```yaml
    - name: Disable swap
      ansible.builtin.command: swapoff -a
      changed_when: false
```

`command` runs a raw shell command — `swapoff -a` turns off all swap devices.

Why `changed_when: false`? `swapoff -a` exits with code 0 whether it did anything or not. Without `changed_when: false`, Ansible would report "changed" every run, even when swap was already off. This overrides the change detection so the playbook output accurately reflects that nothing happened on reruns.

Why swap must be off: the kubelet assumes all memory is real RAM. If a pod uses more memory than its limit, the kubelet wants to OOM-kill it. If swap is available, Linux pages memory to disk instead — the kubelet can't detect this, resource accounting breaks, and scheduling becomes unpredictable. The kubelet will refuse to start if it detects swap.

#### Task 2: Remove swap from fstab

```yaml
    - name: Remove swap from /etc/fstab
      ansible.posix.mount:
        path: swap
        state: absent
```

`ansible.posix.mount` manages entries in `/etc/fstab`. `state: absent` removes any swap mount entry. `path: swap` matches fstab lines where the mount point is literally "swap" (e.g., `/swapfile swap swap defaults 0 0` or `UUID=... none swap sw 0 0`).

This is idempotent by design — if no swap entry exists, it does nothing. The reason this is a separate task from `swapoff -a`: without removing the fstab entry, swap re-enables on reboot. Disabling swap requires both immediate action (`swapoff -a`) and persistent change (remove from fstab).

#### Task 3-4: Load kernel modules

```yaml
    - name: Load overlay kernel module
      community.general.modprobe:
        name: overlay
        state: present

    - name: Load br_netfilter kernel module
      community.general.modprobe:
        name: br_netfilter
        state: present
```

`modprobe` loads kernel modules. `state: present` is inherently idempotent — if the module is already loaded, it's a no-op.

**`overlay`** — the overlay filesystem. This is how container images work. Each layer of a Docker/OCI image is an overlay layer, and the container runtime stacks them together. containerd requires this kernel module. Without it, containers can't start.

**`br_netfilter`** — bridge netfilter. When Linux bridges network interfaces (which Kubernetes does for pod networking), traffic crossing the bridge normally bypasses iptables. Loading `br_netfilter` makes iptables see bridged traffic. Without this, Services and NetworkPolicy silently fail because kube-proxy's NAT rules in iptables don't apply to pod traffic.

#### Task 5: Sysctl configuration

```yaml
    - name: Create sysctl config for Kubernetes
      ansible.builtin.copy:
        dest: /etc/sysctl.d/99-kubernetes.conf
        content: |
          net.bridge.bridge-nf-call-iptables = 1
          net.ipv4.ip_forward = 1
        mode: "0644"
      notify: reload sysctl
```

Drops a file into `/etc/sysctl.d/`. Files in this directory are loaded by `sysctl --system` at boot time, alphabetically. The `99-` prefix ensures this loads last (higher numbers override lower numbers).

**`net.bridge.bridge-nf-call-iptables = 1`** — the runtime counterpart to loading the `br_netfilter` module. The module provides the capability; this sysctl actually enables it. When set to 1, packets crossing a Linux bridge are processed by iptables rules. Kubernetes depends on this for kube-proxy to create DNAT rules that make Services work.

**`net.ipv4.ip_forward = 1`** — enables IP forwarding. A Kubernetes node acts as a router: it forwards packets from a pod on its bridge to the node's main interface, then out to another node. Without IP forwarding, a pod on node A can't reach a pod on node B. Linux disables forwarding by default because a machine that isn't a router shouldn't forward.

**`notify: reload sysctl`** — tells Ansible "if this task actually writes or changes the file, trigger the handler called `reload sysctl`." Handlers run once at the end of the play, and only if at least one notifying task changed something. The idempotency key insight: on first run, the file is created → handler fires. On second run, the file already matches → task is "ok" → handler doesn't fire.

#### Task 6: Install containerd

```yaml
    - name: Install containerd
      ansible.builtin.apt:
        name: containerd
        state: present
        update_cache: true
```

`state: present` means "install if missing, do nothing if already installed." `update_cache: true` runs `apt-get update` before installing — necessary on a fresh Ubuntu instance where the package cache is stale.

containerd is the container runtime. Kubernetes doesn't run containers directly — it talks to a CRI (Container Runtime Interface) implementation. containerd is the default choice after Kubernetes deprecated dockershim in 1.24.

#### Task 7-10: Configure containerd

```yaml
    - name: Create containerd config directory
      ansible.builtin.file:
        path: /etc/containerd
        state: directory
        mode: "0755"
```

Ensures `/etc/containerd/` exists. The Ubuntu package might create it, but this guarantees it and fixes permissions if they diverge. This is a defensive idempotent guard.

```yaml
    - name: Generate default containerd config
      ansible.builtin.command: containerd config default
      register: containerd_config
      changed_when: false
```

Runs `containerd config default` which prints the complete default containerd configuration to stdout — several hundred lines of TOML. `register` captures that output into the variable `containerd_config.stdout`. `changed_when: false` because this command only reads, never writes.

```yaml
    - name: Write containerd config with SystemdCgroup enabled
      ansible.builtin.copy:
        dest: /etc/containerd/config.toml
        content: |
          {{ containerd_config.stdout | replace('SystemdCgroup = false', 'SystemdCgroup = true') }}
        mode: "0644"
      notify: restart containerd
```

Takes the full default config as a string, performs a text replacement on the one line that matters, and writes it. The Jinja2 `replace()` filter does a simple string substitution — finds `SystemdCgroup = false` and changes it to `SystemdCgroup = true`.

The two-step approach (generate → modify → write) exists because there's no Ansible module for "edit containerd config." This pattern gives you a complete valid config with a single targeted change, rather than trying to maintain a full config template that could drift from containerd's defaults.

**Why SystemdCgroup?** Cgroups (control groups) are how Linux limits CPU, memory, and I/O for processes. Two cgroup drivers exist:
- **cgroupfs** — manages cgroups directly through the filesystem. This is containerd's historical default.
- **systemd** — delegates cgroup management to systemd. This is the kubelet's default.

When they disagree, both drivers try to manage cgroups independently, leading to:
- Double accounting (both systemd and cgroupfs count the same resources)
- Kubelet warnings: `cgroup driver mismatch`
- Potentially broken resource enforcement (OOM killer can't correctly identify which process exceeded limits)

Setting containerd to use systemd aligns it with the kubelet. They share the same cgroup tree under systemd, and resource limits work correctly.

`notify: restart containerd` triggers a restart, but only if the config file actually changed.

#### Task 11-12: Kubernetes APT repository

```yaml
    - name: Add Kubernetes APT key
      ansible.builtin.get_url:
        url: https://pkgs.k8s.io/core:/stable:/{{ kubernetes_apt_version }}/deb/Release.key
        dest: /etc/apt/keyrings/kubernetes.asc
        mode: "0644"
```

Downloads the GPG signing key from the Kubernetes package repository and saves it to `/etc/apt/keyrings/`. This key cryptographically verifies that packages you download are actually from the Kubernetes project and haven't been tampered with.

`/etc/apt/keyrings/` is the modern secure location for per-repository keys (since Debian 11 / Ubuntu 22.04). The old practice was dumping keys in `/etc/apt/trusted.gpg.d/`, but that made every key in that directory trusted for all repositories — a security problem. The modern approach is per-repo keys with the `signed-by` directive in the sources list.

```yaml
    - name: Add Kubernetes APT repository
      ansible.builtin.apt_repository:
        repo: "deb [signed-by=/etc/apt/keyrings/kubernetes.asc] https://pkgs.k8s.io/core:/stable:/{{ kubernetes_apt_version }}/deb/ /"
        state: present
        filename: kubernetes
```

Writes a `.list` file in `/etc/apt/sources.list.d/kubernetes.list`. The `[signed-by=...]` option pins this repo to the specific key — APT will only trust packages signed with that exact key.

`https://pkgs.k8s.io/` is the community-owned package repository. Since August 2023 (Kubernetes 1.28), Google stopped hosting `apt.kubernetes.io` and the new repo is managed by the Kubernetes community under `pkgs.k8s.io`.

#### Task 13-14: Install and hold Kubernetes packages

```yaml
    - name: Install kubelet, kubeadm, kubectl
      ansible.builtin.apt:
        name:
          - kubelet
          - kubeadm
          - kubectl
        state: present
        update_cache: true
```

Three packages:
- **kubelet** — the node agent that runs on every node. It receives PodSpecs from the apiserver and ensures the described containers are running. This is the lowest-level Kubernetes component on each node.
- **kubeadm** — the bootstrap tool. Used for `kubeadm init` (creates the control plane) and `kubeadm join` (adds workers). Not a daemon — it's a CLI that runs once to set up or join.
- **kubectl** — the CLI client. Useful on any node for debugging and running `kubectl get nodes` without installing it locally.

```yaml
    - name: Hold kubelet, kubeadm, kubectl versions
      ansible.builtin.dpkg_selections:
        name: "{{ item }}"
        selection: hold
      loop:
        - kubelet
        - kubeadm
        - kubectl
```

`dpkg_selections` sets the APT hold state. A "hold" tells APT: "never upgrade this package." Without this, a routine `apt upgrade` could silently bump k8s from 1.36 to 1.37, and the rest of the cluster won't work.

Kubernetes has a strict version skew policy: the kubelet can be at most N-2 minor versions behind kube-apiserver, and kubectl is N±1. Bumping one node's kubelet ahead of the others breaks this. The hold prevents accidental upgrades — you control when and how to upgrade the cluster.

`loop` iterates the three packages, running the task three times with `{{ item }}` set to each.

#### Handlers

Handlers are special tasks that run only when notified, and only once, at the end of the play:

```yaml
  handlers:
    - name: reload sysctl
      ansible.builtin.command: sysctl --system
      changed_when: true
```

`systctl --system` reloads all sysctl config files (`/etc/sysctl.d/*`, `/etc/sysctl.conf`, etc.) and applies them to the running kernel. `changed_when: true` means it always reports "changed" when it fires (it's actually modifying kernel parameters, so that's accurate).

Why a handler? The config file task only notifies if it wrote the file. If the file already exists with the right content, the handler doesn't fire — no pointless sysctl reload. Also, even if multiple tasks notified this handler, it only runs once.

```yaml
    - name: restart containerd
      ansible.builtin.systemd_service:
        name: containerd
        state: restarted
        enabled: true
```

Restarts containerd — only fires if the config file actually changed (the task that writes `config.toml` notifies it). `enabled: true` ensures the service starts on boot.

Without a handler, you'd need an explicit "restart containerd" task that always runs — wasteful on every playbook execution. The handler pattern makes the playbook truly idempotent: if the config didn't change, containerd isn't needlessly restarted.

### `inventory.ini`

The live inventory, auto-generated by Terraform during `terraform apply`. Gitignored because it contains real IP addresses.

Structure generated by `inventory.tftpl`:
```ini
[all:vars]
ansible_user=ubuntu
ansible_ssh_private_key_file=~/.ssh/id_ed25519
ansible_python_interpreter=/usr/bin/python3

[master]
master ansible_host=54.12.34.56

[workers]
worker-1 ansible_host=54.78.90.12
worker-2 ansible_host=54.78.90.13

[k8s:children]
master
workers
```

The `[k8s:children]` group is a meta-group — `hosts: k8s` includes both master and workers. The `setup.yml` playbook uses `hosts: all` regardless, but if you later write playbooks doing `hosts: master` (for `kubeadm init`) or `hosts: workers` (for `kubeadm join`), this grouping is how you'd target them.

### `inventory.ini.example`

A template showing the expected inventory format. New users reference this to understand how Ansible discovers its targets. The `<MASTER_PUBLIC_IP>` and `<WORKER1_PUBLIC_IP>` placeholders make the format self-documenting.
